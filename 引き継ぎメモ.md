> 🆕 **2026-06-29（v0.7.4）＝ PitType 流れる速さをフレームレート非依存に修正（高リフレッシュモニターで速すぎた不具合）**。ゆうた「流れてくるスピード速くなってない？」。
> - **原因**：`typing.html` の `loop()` が `c.x-=px/60`（1フレームで speed/60 px進む＝**60fps前提**）。120Hz/144Hzモニターでは rAF が毎秒120/144回呼ばれ、**実速度が約2.0〜2.4倍**になっていた（フレームレート依存のバグ）。v0.7.3 の `?v` 更新でキャッシュが切れ最新版が本来のレートで回り出し顕在化。
> - **修正**：実時間ベースに変更。`loop(now)` 先頭で `dt=G.lastT?Math.min((now-G.lastT)/1000,0.05):1/60; G.lastT=now;`（タブ復帰時の飛びは0.05sでクランプ）。移動を `c.x-=px*dt` に。`beginLoop()` で `G.lastT=0` 初期化。これで **60/120/144Hz いずれでも一定（狙い ≒ speed px/秒）**。難易度(持ち時間)ごとの `speed`・`TA_LV.sp`・初心者0.62倍などの倍率はそのまま効く。
> - **検証**：nodeで旧/新を1秒シミュレート＝旧 60fps:56 / 120fps:110 / 144fps:132 px/s（fps依存）→ 新は全fpsで 55〜56 px/s（狙い55）に一定化を確認。
> - **配線**：`index.html` の iframe を **`typing.html?v=14`**（旧 v13）。**同期**：スタンドアロン版 `PitType/index.html`（タイムアタック前の旧エンジンだが loop は同型）にも同じ修正を適用済み。
> - ⚠ push は PowerShell（bash/gitは大きいHTMLを切り詰める）。

## 🚀 公開手順（初回・PowerShell）
GitHubで空のリポジトリ `CoreTools` を作ってから：
```
cd D:\アプリ開発\CoreTools
git init
git add -A
git commit -m "v0.1.0: CoreTools 初版（単位変換・和暦西暦）"
git branch -M main
git remote add origin https://github.com/yuta19kmail-coder/CoreTools.git

```
その後 GitHub の Settings > Pages で main / (root) を公開に。以後の更新は `git add -A` → `commit` → `push` だけ。
